## [0.1.0-alpha.8] — 2026-05-25

### Changed
- **Vault search guidance now covers *drilling into* a large page, not just *finding* one — closes a token blind spot measured at 47% of all `vdd_bash` tokens.** A 95-session token audit of a real project (`locator`) showed VDD costs only ~5% of session context (median), and `vault_read` is justified (a layer page is a **15.8×** compression of the code its `code_refs:` point to). But within VDD's own footprint, the largest avoidable chunk (97K tokens, 47% of Bash output) was the model firing broad `grep -n "A\|B\|C…"` alternations at *large* pages (`positioning.md` 33KB, `decision-log.md` 190KB) to extract a section — dumping 30–40 full prose lines (~5K tok/call). The existing MCP-default guidance (`CLAUDE.md` + skill routing tables) only addressed page-*finding*, which already used the cheap `grep -rliF -l` (filenames only); the page-*drilling* moment had no guidance. Added the missing case: `vdd-workflow` search table gains a canonical "pull a section/keyword from a known large page" row (→ `search_notes` ranked snippet OR `Read` + `offset`/`limit`, never broad `grep -n`), with one-line in-place directives in `vdd-investigate` (Phase 2 page read), `vdd-plan` (HARD-GATE read — its multi-step modes re-check pages most), and `vdd-explain` (Quick reference, step 3b). Guidance-only; effect to be re-measured before any enforcement gate is justified.

## [0.1.0-alpha.7] — 2026-05-23

### Fixed
- **V-09 (Check 13) is now notation-agnostic — no more false drift warnings when a kebab vault key is backed by a PascalCase/camelCase code symbol.** The contract→code matcher grepped the key's leaf token (`distances-received`) as a **case-sensitive, separator-literal** string, so a legitimately-backing symbol in another notation (`DistancesReceived`, `distancesReceived`, `distances_received`) never matched and V-09 falsely warned that the code didn't back the key. Since VDD's convention is lowercase/kebab keys while most languages name symbols in PascalCase/camelCase, this misfired on essentially every cross-notation binding. Fix: build the search pattern as `<seg>[-_]?<seg>…` (each kebab/snake separator becomes an optional separator) and match case-insensitively (`grep -qiE`). V-09 verifies the key's *concept* is backed by code, not that the spelling is identical — this matches that intent. Genuine drift (the concept's words absent from the code entirely) still warns; the contiguous-segment requirement keeps false-negatives minimal. WARN-only as before (never blocks V-06). Verified: `distances-received` ↔ `DistancesReceived` passes, `phase-verified` ↔ `PhaseVerified` passes, an absent concept still warns; `dev/test` 13/13, `check-counts` green.

### Changed
- **The `docs/vault/_lint.sh` wrapper is gone — lint runs via `vdd lint` like every other tool. Zero plugin-owned files copied per-project. BREAKING.** alpha.5 made the schema / CONSTRAINTS / examples plugin-owned, leaving the `_lint.sh` locator wrapper as the *only* plugin-owned file still written into a project — the odd one out, since map / blast / impact / schedule / bootstrap / grasp-gate had all moved to the `vdd` dispatcher on PATH. `vdd lint` execs the exact same plugin `scripts/_lint.sh` the wrapper did (both resolve the vault from `$CLAUDE_PROJECT_DIR` / `$PWD` — byte-identical behavior), so the wrapper was pure redundancy. Removed: `install.sh` no longer writes `docs/vault/_lint.sh` (the `WRAPPER_LINT` heredoc + `write_wrapper` helper are deleted); the legacy wrapper from older installs is **relocated to `_archive/_migrated-<ts>/`** alongside the other migrated plugin-owned files. **Net effect: `docs/vault/` now holds only `index.md` + the user's layer pages — nothing plugin-owned.** Every `bash docs/vault/_lint.sh` reference swept to `vdd lint` across all workflow skills (`vdd-review` / `vdd-done` / `vdd-plan` / `vdd-build` / `vdd-init` / `vdd-investigate` / `vdd-log` / `vdd-onboarding` / `vdd-workflow`), `scaffold/CLAUDE.md` (Hard rule 7 + lint-doc) and `scaffold/CONSTRAINTS.md` (V-06), `scaffold/index.md`, `README`, `docs/QUICKSTART`, `docs/INTEGRATION`, `dev/MIGRATION`, `CONTRIBUTING`, `agents/vault-planner` + `agents/vdd-implementer`, and the `vdd-blast` / `bootstrap` error hints. `smoke.yml` flips from asserting the wrapper exists to asserting **no** plugin-owned file leaks into the project, and adds a migration test (legacy files → `_archive/_migrated`, lint still clean). **Tradeoff**: lint now requires the `vdd` dispatcher on `PATH` — automatic inside Claude Code, but a bare terminal / CI must prepend the plugin's `bin/` (or call `scripts/_lint.sh` directly with `CLAUDE_PROJECT_DIR`). The deeply-referenced by-hand `bash docs/vault/_lint.sh` contract is retired in favor of `vdd lint`. **Migration**: existing projects drop the wrapper on the next `install.sh` / `/vdd-init` run; use `vdd lint` (Claude Code) or `PATH="<plugin>/bin:$PATH" vdd lint` (plain shell).
- **Fixed a bash 3.2 parse break in `install.sh`** introduced while editing the `ROOT_HINT_BLOCK` heredoc: macOS bash 3.2's `$(cat <<'EOF' … EOF)` scanner counts single-quotes in the here-document body, so a lone apostrophe ("the plugin's lint") flipped quote state and aborted the script at the `$(` with `unexpected EOF`. Reworded to drop the apostrophe; `bash -n` clean on 3.2.

### Changed
- **Schema + CONSTRAINTS V-XX + examples are plugin-only — no longer copied per-project. BREAKING.** After alpha.4's `vdd` dispatcher (tools on `PATH`) and the `template/` → `scaffold/` split, the last plugin-owned files still copied into every project were `docs/vault/CLAUDE.md` (the universal vault schema), `docs/vault/CONSTRAINTS.md` (the V-XX invariants), and `docs/vault/examples/`. All three are identical across projects, yet `CLAUDE.md` was `copy-once` + preserve-if-exists (so it **froze at first install** and never picked up plugin updates) and `examples/` was refresh-on-install (stale until the next `install.sh` re-run). Both are now read straight from the plugin: the **SessionStart hook injects their absolute plugin-cache paths** into context every session (path-injected, not content-injected — the Read-on-demand lazy-load + the ~9–12KB session budget are preserved), and **examples** are read from the plugin cache by `vdd-onboarding`. Same mechanism as the `_lint.sh` locator wrapper and the `vdd` dispatcher: a project holds nothing plugin-owned, so a plugin update applies immediately with **no `install.sh` re-run**. The only plugin-owned file remaining in a project is the stable `_lint.sh` locator wrapper (content never changes). **C-XX (project-specific rules) is retired** — `CONSTRAINTS.md` is now VDD-universal V-XX only; a project incident that should harden behavior becomes a new V-XX upstream or a page `decisions:` entry. `_lint.sh` 4c (CONSTRAINTS ↔ lint coverage) becomes a **plugin-internal self-consistency check** (both the V-XX headings and the lint markers ship in the plugin — V-XX 11/11). Migration: legacy per-project `CLAUDE.md` / `CONSTRAINTS.md` / `examples/` are **relocated to `_archive/_migrated-<ts>/`** on the next `install.sh` run — a dir every vault tool already excludes, so they never pollute the graph or trip lint, yet a project that had grown `C-XX` rules keeps them recoverable there. `.removed.*` / `.bak.*` artifacts are now excluded by `_lint.sh` / `vdd-map` / `vdd-blast` / `vdd-schedule` / `vdd-plan` (closes a `vdd-map` phantom-node bug where a migrated `examples.removed.<ts>/` folder rendered as an isolated layer). Doc/skill sweep: `vdd-plan` / `vdd-review` / `vdd-done` / `vdd-workflow` / `vdd-onboarding` / `vdd-init` (CONSTRAINTS read → plugin-injected path; `vdd-onboarding` install sentinel `test -f docs/vault/CLAUDE.md` → `test -d docs/vault`), `README`, `docs/INTEGRATION`, `docs/QUICKSTART`, `agents/vdd-implementer`, `scaffold/index.md`. **Removed machinery** (net −252 lines): the hook's §0 V-XX hash-sync (~190 lines: marker-block replace, `v_hash` frontmatter pin, legacy-file auto-migration), `_lint.sh` 4c-bis V-XX drift detection, and `install.sh`'s CLAUDE.md / CONSTRAINTS / examples seed-copy + the `refresh_file` helper. **Migration**: `docs/vault/CLAUDE.md` and `docs/vault/CONSTRAINTS.md` no longer exist in projects — Read the plugin paths the SessionStart hook injects; existing projects auto-migrate on the next `install.sh` / `/vdd-init` run.
- **`template/` dissolved → standard `scripts/` + `scaffold/` layout.** `template/` had been doubling as both the plugin's runtime tool directory *and* the per-project scaffold source — a non-standard mix (standard plugins keep executables at a root `scripts/` / `bin/` and don't carry a "files copied into a user project" folder). Split it: the deterministic tools (`_lint.sh`, `vdd-*.sh`, `bootstrap.sh`, `grasp-gate.sh`, `vdd-yaml-lib.sh`) → root **`scripts/`** (where the `vdd` dispatcher and the `_lint.sh` wrapper resolve); the per-project scaffold (`CLAUDE.md`, `CONSTRAINTS.md`, `index.md`, `examples/`) → **`scaffold/`**; `template/` removed. Git renames preserve history. Path sweep across `install.sh` (copies from `scaffold/`, lint wrapper → `scripts/_lint.sh`), `bin/vdd`, the SessionStart hook (reads `scaffold/CONSTRAINTS.md`), `_lint.sh` internals (the sibling `vdd-yaml-lib.sh` was `$LIB_DIR/scripts/…` — now `$LIB_DIR/…`; the plugin-SoT CONSTRAINTS for the V-XX drift check moved to `$LIB_DIR/../scaffold/`), `dev/check-counts.sh`, `dev/test/run.sh`, and `smoke.yml` (install assertions rewritten for the dispatcher model: no per-project `docs/vault/scripts/*.sh` wrappers, `bin/vdd` existence + `vdd lint` delegation checked). `dev/test` 13/13; `check-counts` green.

## [0.1.0-alpha.4] — 2026-05-23

### Added
- **V-10 review freshness + `_lint.sh --stamp` (closes the two weak links left after the hook removal).** Two failure modes survived the move to "enforcement = CONSTRAINTS V-XX + skill procedure": (1) `vdd-review` was a *dotted line* — whether a page's intent was actually re-checked after the code moved was unverifiable and unrecorded; (2) `decisions:` frontmatter only grew — nothing pushed stale rationale toward the archive pattern, so long-running pages decayed into archaeology. Both are now mechanical, with **zero new infrastructure** (bash + the existing `shasum`/`awk` lint primitives — no test runner, no agent dispatch, no git dependency):
  - **`_lint.sh --stamp [page]`** writes `reviewed_code_hash` (a content fingerprint — aggregate sha256 over the page's resolving `code_refs` files, path-labelled + locale-sorted so it is clone/rebase/machine-independent) + `reviewed_at` into the page frontmatter via byte-level field replace/inject (no YAML re-emit — honors the frontmatter-write Hard Rule). Bare `--stamp` stamps every page.
  - **Check 14 (V-10)** recomputes the fingerprint and **WARNs on mismatch** ("intent may be stale — re-review then re-stamp"). Content-based, so a stamp is fresh *immediately* after review (no commit needed) and survives clone/rebase — deliberately chosen over a git-sha receipt, which lagged the review-before-commit flow and tripped on rebase. WARN-only: a content change is a re-review prompt, never a V-06 block; renames/whitespace trip it by design (cheap to clear). Pages with code_refs but no stamp are summarized once (INFO), never an error.
  - **Check 12b (V-08 decay)** WARNs when a page's active `decisions:` exceed a count cap (`DECISIONS_COUNT_CAP=25`) — nudging the existing `<layer>/decision-log.md` archive pattern before the page becomes unscannable.
  - **Skill wiring**: `vdd-review` Mode B (Analyze) Step 5 auto-stamps every page it verdicts CONSISTENT (and *only* those — DRIFT/UNVERIFIABLE pages keep their flag raised); `vdd-done`'s CONSTRAINTS gate adds V-10 (no stale WARN for this session's pages before the commit message; the stamp edits ride along in the commit). The binding intent verdict stays human/`vault-verifier`; Check 14 is its deterministic prompt.
  - Lint check total **13 → 14** (`README` / `docs/QUICKSTART` / `docs/INTEGRATION` count sites updated; `dev/check-counts.sh` green). CONSTRAINTS V-XX block gains `### V-10` (the SessionStart hash-sync propagates it to installed projects; `_lint.sh` is sourced from the plugin path so the new checks are live on next lint run — no per-project copy). `template/CLAUDE.md` schema documents the two optional `reviewed_*` fields.
- **`vdd-live` — opt-in real-time session dashboard (`tools/vdd-live/`).** A read-only live view of a VDD session: phase bar (`plan → build → review → done`), active skill, dispatched agents + completed-by-type tally, the planned `vdd-schedule` batch grid, main-thread vault reads/writes, and script/lint status. Parts: `vdd-live-lib.sh` (shared path resolver), `vdd-live-emit.sh` (hook — classifies each event to one JSONL line, `flock`-appends, always exit 0 / no stdout / `jq`-absent-safe, strips the `vault-driven-development:` skill namespace), `vdd-live-reduce.sh` (full one-screen dashboard), `vdd-live-watch.sh` (`tail -f` re-render loop), `vdd-live-status.sh` (one compact line `VDD build · impl 2/3 · lint ✓` for the Claude Code `statusLine` — same window, no extra terminal). **Zero repo footprint**: event state is written under `${CLAUDE_CONFIG_DIR:-~/.claude}/vdd-live/<encoded-project>/events.<session_id>.jsonl` (mirrors Claude's own per-project transcript storage), NOT in the user's repo — no `.gitignore` step, no commit risk, and per-session files keep concurrent windows from interleaving. **Not wired into `install.sh` and adds no shipped hook** — the user merges `settings.snippet.json` (hooks + optional `statusLine`) into one project's settings to opt in; the plugin's "one hook — SessionStart" rule is untouched. **Platform-imposed scope** (confirmed against Claude Code hooks docs): a Task subagent's *inner* tool calls do not fire the parent session's hooks (context isolation), so the dashboard sees the orchestration layer — skill routing, dispatches, `SubagentStop` (carries `agent_type`, hence a count per type not a per-page checkmark), main-thread vault/script activity — but a parallel implementer's per-edit work is a black box between dispatch and stop. `owner_page` is parsed heuristically from the dispatch prompt. The statusLine slot holds one command, so vdd-live's line conflicts with another statusline plugin (e.g. claude-hud) — pick one.
- **`vdd-schedule.sh` — deterministic parallel-batch scheduler (`template/scripts/`).** `vdd-build`'s parallel-vs-sequential decision was eyeballed by the main LLM ("≥3 independent owner_page groups, no inter-group deps") — a judgment call blind to the vault's own cross-layer contract edges, so a `broadcaster → reactor` dependency could be missed and two contract-coupled pages dispatched concurrently. The new script reads the broadcast graph (`broadcasts` / `reacts_to` / `emits_to` / `intent_refs`) **restricted to the owner_page set**, derives precedence edges (broadcaster before reactor/emitter; intent source before dependent), and topologically layers the set into parallel-safe batches via Kahn's algorithm (awk; bash-3.2 safe, no `_reverse-index` / git dependency). Emits a human report + a machine block (`SCHEDULE_BATCH n p…`) consumed by `vdd-build`, which now dispatches **batch-by-batch**: concurrent within a batch (single-message when width ≥3), barrier between batches. A contract cycle among the pages is reported as `SCHEDULE_CYCLE` + exit 3 → routes to `vdd-plan` Mode C instead of dispatching. Wired into `vdd-build` § Decision 2 / Dispatch procedure, `install.sh` wrapper list, `smoke.yml` existence checks, and `dev/test/run.sh` (golden `schedule-3page` + `schedule-cycle`, exit-code assertions for cycle→3 and unresolved-page→1; cycle fixture under `dev/test/fixtures/schedule-cycle/`).
- **Deterministic-script golden tests (`dev/test/`).** The graph-computation core (`vdd-blast`, `vdd-impact`, `vdd-plan`, `vdd-yaml-lib`, `_lint`) had no fixture tests — a refactor could silently break broadcast-graph / blast-radius / intent_refs-closure computation while CI stayed green (`smoke.yml` ran only `bash -n` + manifest + count checks). Added `dev/test/run.sh`: golden-output diffs for `vdd-blast` / `vdd-impact` / `vdd-plan --diff` / `vdd-yaml-lib` extractors against `dev/test/golden/`, plus `_lint` exit-code assertions (clean fixture → 0; dangling-reactor and missing-`code_ref` fixtures → 1). `_lint` is exit-code only because its stdout carries `updated: $(date)` rollup stamps + a time-relative staleness check (not reproducible). Fixtures under `dev/test/fixtures/` — a 2-layer sample vault + a `plan-delta` old/new pair + two broken vaults. Regenerate goldens with `UPDATE_GOLDEN=1 bash dev/test/run.sh` (review the diff — never blind-update). Wired into `smoke.yml`. Same regression-lock philosophy as `dev/check-counts.sh`.
- **`vdd-map` — frontmatter-derived broadcast/intent graph + `/vdd-map` skill (`template/scripts/vdd-map.sh`).** Renders the whole signal graph from the vault's `broadcasts` / `reacts_to` / `emits_to` / `intent_refs` — the same data `vdd blast` traverses for a single key, but for the entire vault. Because it derives from the lint-enforced frontmatter it can never be staler than the vault (no separate code-scan / sync, unlike code-derived knowledge graphs). Modes: **collapsed** (layer nodes, cross-layer edges aggregated — the teaching view, signal-silent layers surfaced as `iso`), **`--raw`** (every broadcast key as a node, unconsumed keys dimmed), **`--orphans`** (plain-text cross-vault V-05 audit — declared keys with zero reactor), **`--layer <n>`**. Identity is `(emitter layer, key)` so a layer page and its `_state-contract` mirror collapse to one node while same-named keys across layers (`loaded`, `initial`, …) stay distinct. The `/vdd-map` skill picks the mode from intent, shows the Mermaid block + a terminal-readable reading (spine / fan-in / anomalies), and hands orphan/silent findings to `vdd-review` for the code-checked verdict — **renders facts, never verdicts**. Mermaid is the external-render format only; the in-page vault convention stays ASCII (`CLAUDE.md` no-Mermaid rule is unchanged — do not paste the map into a page). Skill count **11 → 12** (10 user-invocable).
- **V-11 silent-layer detection (`_lint.sh` Check 4e, the 15th check) + `vdd-review` Analyze triage step.** Drawing the map exposed a gap class V-02/V-05 are structurally blind to: a layer that *should* participate in the graph but declares nothing (e.g. a native bridge that produces events with an empty `broadcasts:`). V-02/V-05 validate only *declared* edges — never the **absence** of an edge that ought to exist. New Check 4e aggregates the four graph fields **per layer** (not per page — `auth/auth.md` alone is empty, but the auth *layer* is connected via `_state-contract.md` + feature pages, so it is not silent) and emits an **INFO** list of fully-silent layers. INFO-only — never WARN, never blocks the V-06 gate (silence is frequently legitimate for cross-cutting layers like `error` / `permission`; a per-layer nag would be noise). It is the deterministic triage input for `vdd-review`. CONSTRAINTS gains `### V-11`; lint check total **14 → 15** (`README` / `docs/QUICKSTART` / `docs/INTEGRATION` count sites + `dev/check-counts.sh` updated; the SessionStart hash-sync propagates V-11 to installed projects). `vdd-review` Mode B gains **Step 1.5 — Triage**: the deterministic floor (lint orphans / silent layers / V-09 / symbol WARNs) hands *facts*; the LLM classifies each into NORMAL / SUSPICIOUS / QUESTION as a **non-authoritative hypothesis** ("look here", not "this is broken") that only orders the `vault-verifier` pass — the FIX-vs-leave verdict stays human, triage labels are never recorded as verdicts, and the classification is deliberately kept OUT of `_lint.sh` (convention heuristics are brittle as deterministic checks). Step 2 reframes the verifier dispatch around the triage hypotheses (settle the specific question against code) rather than "verify the layer".

## [0.1.0-alpha.3] — 2026-05-18

### Changed
- **`${CLAUDE_PLUGIN_ROOT}/template/...` SoT was wrong — replaced with thin wrapper layer in `docs/vault/`.** The previous `alpha.2` release moved all tooling exclusively to the plugin cache and rewrote every caller (124 matches / 28 files) to `bash ${CLAUDE_PLUGIN_ROOT}/template/_lint.sh`. **Two empirical findings invalidated that approach:**
  1. **Claude Code does NOT export `$CLAUDE_PLUGIN_ROOT` to the Bash tool environment** — running `bash ${CLAUDE_PLUGIN_ROOT}/template/_lint.sh` from a skill body resolves to `bash /template/_lint.sh` and fails.
  2. **Claude Code does NOT substitute `${CLAUDE_PLUGIN_ROOT}` in SKILL.md bodies** — the placeholder reaches the LLM literal, forcing the LLM to either (a) pass through and fail (per 1) or (b) pre-resolve via guessing (typically hitting the developer's local clone path, breaking portability).
  Verified by inspecting other published plugins (`claude-mem@thedotmack`, `claude-hud@claude-hud`) — neither uses `${CLAUDE_PLUGIN_ROOT}` inside SKILL bodies. `claude-hud`'s `statusLine` command uses a runtime locator (`ls -td .../plugins/cache/.../*/ | tail -1`). That is the working pattern.

  **New design**: `install.sh` writes thin wrapper files at `docs/vault/_lint.sh` and `docs/vault/scripts/*.sh`. Each wrapper:
  - locates the latest installed plugin cache via `ls -td "${CLAUDE_CONFIG_DIR:-$HOME/.claude}"/plugins/cache/pdev-jay/vault-driven-development/*/ | head -1`,
  - delegates via `exec bash "${PLUGIN_ROOT}template/<script>" "$@"`,
  - has zero plugin logic (5-12 lines) — content never drifts across plugin updates.

  **Effect**: user-facing contract restored to `bash docs/vault/_lint.sh` / `bash docs/vault/scripts/<name>.sh` — no `${CLAUDE_PLUGIN_ROOT}` dependency anywhere LLMs or shells touch. Plugin updates still take effect immediately because the wrapper's `*/` glob picks the latest version on every invocation. The wrapper file itself never changes unless the marketplace name (`pdev-jay`) or plugin name (`vault-driven-development`) changes.

  **Callers swept back**: 28 files, all `${CLAUDE_PLUGIN_ROOT}/template/_lint.sh` → `docs/vault/_lint.sh` and `${CLAUDE_PLUGIN_ROOT}/template/scripts/` → `docs/vault/scripts/`. Hook regex stays basename-based (`grep -F 'vdd-blast.sh'`) so any invocation path still registers as evidence.

  **CI smoke.yml**: now asserts (a) wrapper files exist + are executable, (b) wrapper file size < 20 lines (heuristic — full scripts are 100+), (c) `vdd-yaml-lib.sh` is NOT in project (plugin-only sourced), (d) plugin source has the real tooling, (e) wrapper-mediated lint passes when a symlinked "ci-version" cache is created.

  **Migration for existing users on `alpha.2`**: just `claude plugin marketplace update pdev-jay && claude plugin install vault-driven-development@pdev-jay --scope user`. The new `install.sh` (when re-run, or auto-run by `vdd-init`) will create the wrappers and archive any stale per-project full scripts.

- **Workflow skills require a per-member impact verdict (`V-07`).** `vdd-spec`, `vdd-plan`, and `vdd-investigate` no longer accept a free-form Impact Analysis — the `vdd-impact.sh` / `vdd-blast.sh` enumerated set must be pasted and *every* member given an `affected` / `unaffected` / `deferred` verdict. Coverage (every member has a verdict) is the skill-checked gate; verdict *correctness* stays the user's review plus the post-hoc `_lint.sh` broadcast-graph / `code_refs` checks. `dev-workflow` lists `V-07` in its `V-XX` rule enumeration.

- Stack-specific vocabulary across skills / agents / template replaced with platform-agnostic wording.

### Fixed
- **Owner-page path matching no longer mis-owns files (`grep -rlwF` → delimiter-bounded match).** `grep -w` treats `/` and `.` as non-word characters, so `lib/auth/login.dart` falsely "word-matched" inside `old/lib/auth/login.dart` and `login.dart.bak`, and a project-root prefix mismatch produced silent misses. This corrupted the owner-page set feeding Tier classification, the `_state-contract` blast gate, V-01, and the `vdd-impact` closure. Fixed in `hooks/pre-tool-use-vault-read-gate.sh`, `hooks/post-tool-use-vault-write-gate.sh`, `hooks/session-start-vault-inject.sh` (delimiter-bounded ERE; hooks stay self-contained), and `template/scripts/vdd-impact.sh` (exact `code_refs` path / directory-prefix compare).
- **One canonical YAML extractor (`scripts/vdd-yaml-lib.sh`) replaces four divergent copies.** `_lint.sh` and `vdd-plan.sh` previously used a block-only extractor that silently emitted **zero items** for inline arrays (`code_refs: [a, b]`) and never stripped quotes — so quoted broadcast keys (`'phase:idle'`, exactly what MCP YAML writers produce) caused false "undefined key" warnings and inline-array pages vanished from V-02/V-03 checks and the reverse-index. All scripts now source the single inline+block+quote+empty-aware implementation; `CLAUDE.md`'s "both styles accepted by `_lint.sh`" is finally true.
- **CRLF / UTF-8 BOM no longer silently zeroes frontmatter parsing.** A CRLF page passed the frontmatter check (grep-tolerant) but every awk parser saw an empty frontmatter (`/^---$/` never matched `---\r`), so the page contributed nothing to `_reverse-index`/`_progress`/`_decisions` with no error. The canonical extractor normalizes CR + a leading BOM before parsing; `_lint.sh` now hard-errors on CRLF vault pages and on duplicate page basenames (Hard Rule 3 — wikilinks resolve by basename, so a collision silently resolved links/closures to the wrong page).
- **`_lint.sh` decisions rollup loss on archive pages.** Pages with empty `decisions: []` frontmatter (the canonical V-08 archive pattern — long rationale moved to a child `decision-log.md` body) previously vanished from the chronological `_decisions.md` rollup, because the rollup only scanned frontmatter `decisions:` arrays. Added a body fallback: when frontmatter `decisions:` is empty, scan body markdown for `## YYYY-MM-DD — title` (or `### YYYY-MM-DD — title`, em-dash) headings and feed them into the rollup. Title is truncated to 120 chars for scannability. Fires only when frontmatter is empty so the V-08 trim discipline isn't undermined — pages keeping frontmatter notes (the normal case) are unaffected; archive pages with body-only history are restored to the rollup. Verified on locator/positioning: 99 archive entries reappear in `_decisions.md` alongside the other layers' frontmatter entries.
- **`_lint.sh` Check 11 (oversized page) false positive on archive child pages.** The check warned every page > 40KB, but its original intent was Rule 0 — "the layer page must remain Read-able under the 25K-token limit." Child pages (zoom 1+) are not Rule 0 candidates, so an intentional 141KB `decision-log.md` archive incorrectly tripped the warning. Narrowed Check 11 to `zoom: 0` layer pages only; archive child pages can grow as large as the host allows.
- **`template/CLAUDE.md` documents the archive child page pattern.** Added a short subsection under "Logging convention" pointing at `<layer>/decision-log.md` (zoom 1, `decisions: []` on the layer page, `## YYYY-MM-DD — title` headings in the archive body) as the canonical V-08 escape valve when a layer accumulates more decisions than can be reasonably trimmed. The fallback rollup in `_lint.sh` is the matching machinery.
- `docs/INTEGRATION.md` and `docs/runtime-flow.md` reconciled to the 4-hook + `minimal`-default model — they previously described 2 hooks, a `standard` default, and a Stop hook the plugin "doesn't ship".
- **`_lint.sh` 4c coverage check false positive.** The `V-XX` / `C-XX` self-grep used `${BASH_SOURCE[0]}` (a relative invocation path) after the script had already `cd`'d into the vault dir, so the grep target never resolved — `covered=0` on every documented `bash docs/vault/_lint.sh` run, producing a permanent false `WARN`. Now greps the absolute `$VAULT_DIR/_lint.sh`.
- **`vdd-plan.sh --staged` / ref modes silently dropped `_state-contract.md`.** The `grep -v '/_'` filter excluded every path containing `/_`, including `docs/vault/<layer>/_state-contract.md` — the primary emitter page the tool exists to track. Contract-delta plans came back empty. Filter now mirrors `--diff` mode: only the four auto-rollups + `examples/` + `_archive/` are excluded.
- **PostToolUse owner-match inconsistent with PreToolUse.** PostToolUse used `grep -rln` (regex, substring) while PreToolUse / `vdd-impact.sh` use `grep -rlwF` (word, fixed-string) — the two hooks could disagree on which page owns a file. Aligned to `grep -rlwF`. Also dropped the `head -3` cap on the match list: it fed `classify_tier`, so a truncated list could hide a `_state-contract.md` owner and misclassify a Tier-A file as Tier-B.
- **Stop hook missed `[BLAST]` / `[MCP-REDIRECT]` block evidence.** The bypass detector grepped `[VAULT-GATE] BLOCKED` literally, so a session stopped only by the `_state-contract` blast gate or the MCP-redirect gate could still false-fire the "code edited without consulting vault" warning. Pattern now matches all three block variants.
- **`bootstrap.sh` survey listed `examples/` as a vault layer.** The existing-layers scan skipped `_archive` and `scripts` but not `examples`, so the read-only examples dir showed up as a layer. Now skipped.
- **Agents had no `tools:` restriction — the "read-only, never modifies pages" guarantee was prose-only.** All 7 agent bodies declare "Edit/Write forbidden" and `docs/INTEGRATION.md` claims "(Read/Grep/Glob/Bash only)", but no agent frontmatter declared `tools:`, so each inherited every tool including `Edit` / `Write`. Added `tools: Read, Grep, Glob, Bash` to all 7 — the no-modification invariant is now enforced, not just promised. The "Optional Obsidian MCP" principle in the 5 affected agents is corrected accordingly: subagents run filesystem-only (MCP read-acceleration is a main-session capability).
- **`_lint.sh` V-05 orphan-broadcast check was claimed but never implemented.** `CONSTRAINTS.md` V-05 and the lint script's own comment both stated `_lint.sh` warns on a `broadcasts:` key with no reactor — but no such check existed (Check 8, despite its "orphan broadcasts" name, only detects a missing `_state-contract.md` file). Added the real reverse check (4d): every declared broadcast key must be referenced by some page's `reacts_to:` / `emits_to:`, else `WARN`. V-05's stated detection mechanism is now real.
- **`vdd-log` and `vdd-done` routed cross-layer decisions to a non-existent `ARCHITECTURE.md`.** Both skills told the user to append decisions to a "Decision Log table in `ARCHITECTURE.md`" — a file VDD never scaffolds and the vault schema never defines; `vdd-log` additionally keyed off a legacy `obsidian:` CLAUDE.md section. Rewrote both to be vault-first: every decision lands in a vault page's `decisions:` frontmatter (layer page, or `_state-contract.md` for contract decisions, or the most-central layer for cross-layer), so `_lint.sh` rolls it into `_decisions.md`.
- **`brownfield-vault-onboarding` and `vault-onboarder` invented a vault-root `_state-contract.md` ("global contract")** not defined by the vault schema (which specifies per-layer `_state-contract.md` only). Reframed: the cross-layer broadcast view is the auto-generated `_reverse-index.md`; onboarding step 7 now verifies wiring via `_lint.sh` + `vdd-blast.sh` instead of authoring a global contract page.
- **`_state-contract.md` `zoom:` inconsistent** — the shipped example used `zoom: 0` (which means "layer index"), `vault-onboarder` drafted `zoom: 2`. Both set to `zoom: 1` (a state-contract is one level below the layer index).
- **`vdd-done` over-claimed "0 warnings required"** — its validation gate and checklist demanded `_lint.sh` 0 warnings, but `CONSTRAINTS.md` V-06 (and `vdd-verify`) only gate on exit 0 / 0 errors, with warnings explicitly triageable. Corrected to match V-06.
- **`dev-workflow` Roles table omitted the wiki agents** — `wiki-querier` / `wiki-promoter` were absent from the table despite the skill description counting "7 vault/wiki agents". Added a "(wiki, optional)" row.
- **`hooks/pre-tool-use-vault-read-gate.sh` shipped without its executable bit.** It was committed `100644` while the other three hooks were `100755`; when Claude Code execs the hook directly per `hooks.json`, the non-executable file produces `Permission denied` (exit 126). Mock-testing never surfaced it — tests invoked the hooks as `bash <script>`, which ignores the bit. Only a live `claude --plugin-dir` session (with `--include-hook-events`) exposed it. Now `100755`.
- **PreToolUse evidence checks failed OPEN — satisfied by a mention, not an execution.** The `_state-contract.md` blast gate (`BLAST_CALLED`), the vault-read floor (`IMPACT_CALLED`), and the `paranoid` per-key blast check all scanned the *entire* session transcript for the script name as a substring. But the transcript also holds tool RESULTS, assistant prose, and SessionStart-injected attachments (claude-mem memory, `dev-workflow` body, `vdd-impact.sh`'s own output) — every one of which names `vdd-blast.sh` / `vdd-impact.sh` without running it. Confirmed live: stripping the real invocations out of an actual transcript still left the `_state-contract.md` gate at exit 0. These checks now match only real `tool_use` Bash `.input.command` entries (extracted via `jq`, already a hard dependency) — the evidence base is execution, not text. This only ever ran in `standard`+ profiles, i.e. for users who explicitly opted into enforcement, so the fail-open silently defeated exactly what they opted in for.
